General

  • All feature branches are created off the develop branch.
  • Releases are marked using release tags and release branches.
  • Rebasing is favored over merging.
  • Feature branches are merged into develop via pull requests. Squash and merge strategy is used to merge the branches.
  • Semantic versioning is used for naming release tags and release branches. 
    Ex. v1.0.0
  • Hotfix branch is created off master and then merged directly into master. The develop branch is then rebased with master. Feature branches are rebased with develop.

Feature branches

  • Are created off the develop branch.
  • Are merged into develop.
  • Are deleted after being merged into develop.
  • Are rebased with latest develop if develop branch is updated.
  • Are code reviewed for code quality and functionality prior to being merged into develop branch.

The develop branch

  • Is used for testing by QA prior to release.
  • Will have feature branches merged into it via pull requests (squash and merge strategy is used).

Testing

  • Testing occurs in the develop branch.
  • On failure, the feature branch causing the issue needs to be rebased with the latest develop branch. Then merged back into develop after the issue is fixed.

Releases

  • All feature branches must be rebased with develop after a new release tag is created.
  • Release tags are used for build/deploy jobs (i.e., Jenkins).
