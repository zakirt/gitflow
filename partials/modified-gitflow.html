General

  • All feature branches are created off the master branch.
  • Releases are marked using release tags in the master branch.
  • Rebasing is favored over merging.
  • QA/feature branches are merged into master via pull requests. Squash and merge strategy is used to merge the branches.
  • Semantic versioning is used for naming release tags. 
    Ex. v1.0.0

Feature branches

  • Are created off the master branch.
  • Are merged into master, or into a temporary qa branch.
  • Are always merged directly into master if they contain technical tasks that do not require QA (i.e., unit tests, README updates, etc...).
  • Are deleted after being merged into master.
  • Are rebased with latest master if master is updated.
  • Are code reviewed for code quality and functionality prior to being merged into QA branch.

QA branch

  • Is optional and is created off the master branch
  • Is used in cases where multiple feature branches need to be tested in unison prior to release.
  • Will be deleted after being merged into the master branch.
  • Will have feature branches merged into it via pull requests (squash and merge strategy is used).

Testing

  • Testing occurs in the QA branch.
  • On failure, the feature branch causing the issue needs to be rebased with the latest QA branch. Then merged back into QA after the issue is fixed.

Releases

  • All feature branches must be rebased with master after a new release tag is created.
  • Release tags are used for build/deploy jobs (i.e., Jenkins).
